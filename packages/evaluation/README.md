# @featuregate/evaluation

Private, runtime-neutral evaluation primitives shared by FeatureGate SDK packages.

This workspace owns flag definitions, evaluation context, conditions, local evaluation, caller
defaults, and evaluation results. It must not depend on networking, storage, Node.js, browser,
framework, or OpenFeature APIs. Public SDK builds bundle the code they use from this package.

Evaluation supports enabled and disabled values, ordered equality-based targeting rules, and
deterministic percentage rollouts. Rollout assignment uses the flag key and targeting key so a
subject remains stable without receiving the same bucket across every flag.

## Rollout compatibility

Rollout assignment is a cross-SDK compatibility contract. Inputs are encoded as:

1. UTF-8 `featuregate-rollout-v1`
2. The flag key's UTF-8 byte length as an unsigned 32-bit big-endian integer
3. The UTF-8 flag key
4. The rollout seed's UTF-8 byte length as an unsigned 32-bit big-endian integer
5. The UTF-8 rollout seed
6. The targeting key's UTF-8 byte length as an unsigned 32-bit big-endian integer
7. The UTF-8 targeting key

The SHA-256 digest's first four bytes are read as an unsigned big-endian integer and mapped to an
integer bucket from `0` through `99,999`. Rollout allocation weights use the same 100,000-unit
scale and are selected as ordered, half-open ranges. Strings are encoded exactly as supplied and
are not Unicode-normalized. Seeds are generated by the FeatureGate service and can be rotated to
intentionally assign subjects to new buckets without changing the flag key.
